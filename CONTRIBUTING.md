## üß™ How to Contribute

### 1. Reporting Issues

If you encounter a bug, unexpected behavior, or confusing output, please include:

- Python version  
- OS (Windows, WSL, Linux distro, etc.)  
- The log file generated by the tool  
- The command used to run the script  
- Steps to reproduce the issue  

Logs are especially helpful ‚Äî the tool is designed to be transparent.

---

### 2. Suggesting Features

Feature ideas are welcome!  
Useful suggestions include:

- improvements to logging  
- CSV export enhancements  
- Windows compatibility  
- command‚Äëline flags  
- performance improvements  
- UX refinements  
- integration ideas for UniPlaySong or Playnite  

If you're unsure whether something fits the roadmap, open an issue ‚Äî discussion is encouraged.

---

### 3. Submitting Pull Requests

#### Before you begin:

- Make sure your work targets the **`pre-release`** branch  
- Check `RELEASES.md` to understand the current version boundaries  
- Follow the existing coding style (simple, explicit, readable)  

#### PR Guidelines:

- Keep changes focused and atomic  
- Include a short description of what the PR does  
- Update `RELEASES.md` if your change affects behavior  
- Add comments where logic may not be obvious  
- Test your changes on at least one real directory of MP3s  

#### Code Style:

- Prefer clarity over cleverness  
- Avoid unnecessary dependencies  
- Use `Path` objects for all filesystem operations  
- Keep worker functions top‚Äëlevel for Windows compatibility  
- Maintain deterministic behavior  

---

## üß™ Testing Expectations

The Python port is designed to run in:

- WSL (recommended for development)  
- Linux  
- Windows (partial support; full support planned)  

When testing:

- Ensure `ffmpeg` and `ffprobe` are available in PATH  
- Try both preview mode and reduction mode  
- Verify logs are generated correctly  
- Confirm parallel workers behave as expected  
- Watch for edge cases (no metadata, low bitrate, tiny files, etc.)
